AUTHENTICATION AND PIN FLOW (NO REGISTRATION):
- Remove any user self-registration/sign-up flows; there must be no registration anywhere.
- Only pre-created users (created by Head Office in Admin) can log in with email+password.
- Always require MFA on every login using one-time codes sent to the registered email address (email OTP), including for external invited users.
- After successful email+password+email-OTP, show a staff-picker screen for the selected pharmacy that lists named staff identities with roles. User clicks their name, then enters their PIN.
- PIN rules: numeric only, 4–8 digits; created and reset only by Head Office; store failed attempts; lock after 3 failures for 10 minutes; generate Head Office alert/notification on lockout.
- Master staff identity exists with PIN 145891 (6 digits) and role Super Admin/Master; master can log in from any IP; store the originating IP address in audit logs for every login.
- Enforce pharmacy IP allowlisting for pharmacy logins server-side; pharmacy users cannot log in outside allowlisted pharmacy IPs. Master can bypass IP allowlist but still requires email OTP every time.
- Once staff PIN is validated, do not ask again during the session. Provide explicit Logout action that logs out the staff session and returns to the staff-picker.

DAILY FIGURES CHANGES (MONTHLY MODEL, REMOVE VARIANCE):
- Remove variance concepts, fields, validation, and UI entirely from Daily Figures.

  - EPS Rx Paid, EPS Rx Exempt
  - EPS Items Paid, EPS Items Exempt
  - Paper Rx Paid, Paper Rx Exempt
  - Paper Items Paid, Paper Items Exempt
- These are running totals; implement display and storage to support "today's running total" and "month-to-date running total" views for each metric. Ensure all modules use a monthly model (month-to-date aggregation).
- Include SSP within the Figures section.
- NHS Prepayment should be a currency field in GBP (amount sold), and FP57 Refund should be a currency field in GBP (amount refunded).

CASHING UP CHANGES (CALCULATED GROSS/ACTUAL, DOUBLE-CHECK VARIANCE, MONTHLY MODEL):
- Inputs staff must enter daily (trading days), all mandatory with default 0:
  - To Be Banked
  - Reading Card
  - VAT Analysis buckets labelled exactly: Standard, Exempt, Zero, 5% VAT
  - Pay Outs as flexible dynamic list (amount + detail)
  - Cash Up Variance (entered by staff as double-check)
- System calculations:
  - Gross Taking = sum(Standard + Exempt + Zero + 5% VAT)
  - Actual Taking = To Be Banked + Reading Card + total Pay Outs
  - System Variance = Gross Taking - Actual Taking
- Validation: block submission unless entered Cash Up Variance equals System Variance; show clear error asking to double-check entries.
- Ensure cashing up is stored per trading day but dashboards and reports show month-to-date aggregations.

REPORTING (CUSTOM AND MONTHLY CASHING UP REPORTS):

- Report outputs include totals: Gross Taking, Actual Taking, Variance, To Be Banked, Reading Card, Pay Outs total, and VAT totals (Standard, Exempt, Zero, 5% VAT) plus counts of late submissions and days with non-zero variance.
- Export report as CSV and PDF; store report metadata in DB; log report generation and downloads in audit logs.
